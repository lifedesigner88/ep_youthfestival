# 레트로 초등시절 친구 만들기 게임 개발 계획서

## 1. 프로젝트 개요

옛날 플래시 게임과 초등학교 시절 공책 귀퉁이에 친구 얼굴을 그리던 손그림 놀이의 감성을 살린 **초등시절 친구 만들기 게임**을 제작한다. 사용자는 빠르게 바뀌는 얼굴 파츠를 보면서 하나의 `멈춤` 버튼을 총 5번 누른다. 누를 때마다 얼굴, 눈, 코, 입, 머리가 차례대로 하나씩 고정되고, 다섯 번째 파츠가 고정되면 추억 속 우리 반 친구 같은 캐릭터를 확인한다.

로그인과 백엔드 없이 브라우저에서 바로 실행하며, 최근 결과와 즐겨찾기는 `localStorage`에 저장한다. 완성된 캐릭터와 닮은 친구의 이름을 입력하면 이름을 이미지 아래에 귀여운 손글씨 느낌으로 합성하고, 완성 카드를 고해상도 PNG로 다운로드할 수 있게 한다.

### 핵심 목표

- 인스타그램 링크에서 들어온 사용자가 설명 없이 바로 플레이할 수 있을 것
- 한 손으로 짧고 반복적으로 즐길 수 있을 것
- 공책, 필통, 칠판처럼 세대를 넘어 공감할 수 있는 초등학교 시절의 기억을 떠올리게 할 것
- 예상하지 못한 우스꽝스럽고 귀여운 얼굴 조합이 만들어질 것
- 같은 얼굴 조합에는 언제나 같은 긍정적인 결과명이 표시될 것
- 친구 이름이 들어간 결과물을 휴대폰에 다운로드해 사용자가 원하는 방식으로 전달할 수 있을 것
- 서버 운영 없이 GitHub Pages에서 배포할 수 있을 것

### 1차 버전에서 하지 않는 것

- 회원가입 및 로그인
- 서버 데이터베이스와 기기 간 기록 동기화
- 온라인 랭킹 및 사용자 간 대전
- 관리자 페이지
- 앱스토어용 네이티브 앱
- 사용자가 직접 이미지를 업로드하는 기능

## 2. 대상 사용자와 이용 상황

### 주요 사용자

- 인스타그램 게시물, 스토리 또는 프로필 링크를 통해 유입된 모바일 사용자
- 설치 없이 짧게 즐길 수 있는 가벼운 게임을 원하는 사용자
- 완성된 결과에 친구 이름을 넣어 이미지로 저장하고 직접 전달하려는 사용자

### 대표 이용 흐름

1. 인스타그램에서 게임 링크를 누른다.
2. `나의 초등학교 시절엔 어떤 친구가 있었을까?`라는 질문을 본다.
3. 기억 속 친구의 캐릭터 모드 카드를 한 번 눌러 바로 게임을 시작한다.
4. 선택한 스타일의 얼굴 파츠가 빠르게 변경되는 모습을 확인한다.
5. 화면 하단의 큰 `멈춤` 버튼을 눌러 얼굴을 고정한다.
6. 같은 버튼을 네 번 더 눌러 눈, 코, 입, 머리를 차례로 고정한다.
7. 완성된 얼굴과 결과명을 확인한다.
8. 완성된 얼굴과 닮은 친구의 이름을 입력한다.
9. 이름이 캐릭터 아래에 합성된 PNG를 다운로드한다.
10. 휴대폰에 저장된 이미지를 원하는 친구에게 직접 전달한다.
11. `다시 하기`를 눌러 새로운 조합을 만든다.

## 3. 게임 규칙

### 캐릭터 모드 선택

시작 질문은 **`나의 초등학교 시절엔 어떤 친구가 있었을까?`**로 사용한다. 이 질문 아래에서 기억 속 친구의 모습을 떠올리듯 모드를 고르게 한다. 누구나 원하는 모드를 선택할 수 있으며, 이 값은 사용자 프로필이 아닌 이번 게임의 조합 방식으로만 취급한다.

| 선택지 | 동작 |
| --- | --- |
| 남학생 모습 | 남학생 스타일의 파츠 세트로 플레이 |
| 여학생 모습 | 여학생 스타일의 파츠 세트로 플레이 |
| 랜덤으로 떠올리기 | 성별 스타일 구분 없이 각 부위에서 두 세트의 모든 파츠를 사용 |

- 모드는 게임 시작 전에 한 번 선택하고 5단계 멈춤 도중에는 변경하지 않는다.
- 남학생 또는 여학생 모습을 선택하면 각 부위가 해당 스타일의 파츠 5개 안에서 순환한다.
- `랜덤`을 선택하면 얼굴, 눈, 코, 입, 머리가 각각 두 스타일의 파츠 10개 전체에서 독립적으로 순환한다.
- 예를 들어 남학생 얼굴형, 여학생 눈, 남학생 코, 여학생 입과 머리처럼 성별 느낌에 제한 없는 조합이 가능하다.
- `랜덤`은 별도의 세 번째 에셋 세트가 아니라 기존 두 세트의 파츠 풀을 합쳐 사용하는 모드다.
- 결과 저장 시 사용자의 성별이 아니라 선택한 `characterMode`와 각 파츠의 고유 ID만 브라우저에 저장한다.
- 시작을 빠르게 할 수 있도록 `랜덤`을 기본 선택 상태로 둘 수 있다.

### 파츠 구성

각 스타일은 아래의 파츠를 각각 5개씩 가진다.

| 파츠 | 스타일별 개수 | 예시 방향 |
| --- | ---: | --- |
| 얼굴형 | 5 | 원형, 사각형, 삼각형, 타원형, 땅콩형 |
| 눈 | 5 | 모범생 눈, 장난꾸러기 눈, 졸린 눈, 울먹이는 눈, 반짝이는 눈 |
| 코 | 5 | 점코, 짧은 선코, 세모코, 둥근 코, 낙서식 콧구멍 코 |
| 입 | 5 | 활짝 웃는 입, 삐죽 입, 앞니가 보이는 입, 놀란 입, 장난스러운 메롱 입 |
| 머리 | 5 | 바가지 머리, 반듯한 가르마, 양갈래, 삐죽 머리, 체육시간 뒤 헝클어진 머리 |

| 구분 | 계산 | 조합 수 |
| --- | ---: | ---: |
| 남학생 스타일 | `5⁵` | 3,125개 |
| 여학생 스타일 | `5⁵` | 3,125개 |
| 두 고정 스타일 합계 | `2 × 5⁵` | 6,250개 |
| 랜덤 모드 | `(5 + 5)⁵` | **100,000개** |
| 랜덤 모드 중 성별 느낌이 섞인 조합 | `100,000 - 6,250` | 93,750개 |

랜덤 모드의 100,000개 안에는 파츠가 모두 남학생 스타일인 3,125개와 모두 여학생 스타일인 3,125개도 포함된다. 따라서 모드 간 중복을 제외한 게임 전체의 고유한 시각 조합 수는 **100,000개**다.

### 결과명 조합 규칙

결과명은 얼굴 조합과 무관하게 다시 뽑는 값이 아니라, 확정된 얼굴·눈·코·입·머리의 고유 ID 5개로 계산한다. 따라서 같은 파츠 조합은 플레이 시점이나 선택한 모드와 관계없이 항상 같은 결과명을 얻는다.

- `학교 장면 10개 × 긍정적인 분위기 10개 × 친구 역할 10개`를 조합해 **1,000개의 결과명**을 만든다.
- 랜덤 모드의 100,000개 시각 조합은 1,000개의 결과명에 정확히 100개씩 배정한다.
- 남학생 모습과 여학생 모습은 각각 3,125개의 시각 조합만 사용하지만, 두 모드 모두 1,000개의 결과명 전체에 도달하며 결과명마다 2~5개 조합이 배정된다.
- 다섯 파츠가 모두 결과명 계산에 영향을 주며, 어느 파츠 하나가 달라져도 결과명 인덱스가 달라진다.
- 결과명 단어 목록의 순서와 계산 규칙은 기존 얼굴의 결과명을 바꾸므로 버전 내에서 고정한다.

구현에서는 남학생 스타일 파츠 인덱스를 `0~4`, 여학생 스타일 파츠 인덱스를 `5~9`로 바꾸고 얼굴부터 머리까지의 숫자를 `f, e, n, m, h`로 둔다. 아래 계산으로 얻은 `nicknameIndex`를 고정된 1,000개 결과명 조합표의 인덱스로 사용한다.

```js
const r = (241 * f + 497 * e + 100 * n + 10 * m + h) % 1000
const nicknameIndex = (137 * r + 211) % 1000
```

이 규칙은 전체 100,000개 조합과 두 고정 스타일의 3,125개 조합을 전수 검사해 위 분포를 유지하는지 검증한다.

- 1차 버전의 전체 파츠 에셋은 `2개 스타일 × 5개 부위 × 부위별 5개 = 50개`다.
- 1차 버전은 하나의 `멈춤` 버튼을 총 5번 누르는 방식으로 진행한다.
- 기본 확정 순서는 `얼굴 → 눈 → 코 → 입 → 머리`로 고정한다.
- 버튼을 누를 때마다 현재 차례의 파츠가 고정되고 다음 차례 파츠 하나만 새로 나타나 변경된다.
- 다섯 번째 입력으로 머리가 고정된 직후 최종 결과 화면을 표시한다.
- 고정된 파츠는 해당 게임이 끝날 때까지 다시 변경되지 않는다.
- 버튼 문구는 `얼굴 멈추기 1/5`, `눈 멈추기 2/5`처럼 다음 행동과 진행도를 함께 표시한다.
- 화면에는 5개 파츠의 진행 표시를 두고 `진행 중`, `고정 완료`, `대기 중` 상태를 구분한다.
- 다시 시작하면 각 파츠의 변경 타이밍을 약간 다르게 설정해 조합이 단조롭게 반복되지 않도록 한다.
- 직전 결과와 완전히 같은 조합이 연속으로 나오면 한 파츠를 다시 선택하는 방식으로 중복 체감을 줄인다.

### 게임 상태

```text
INTRO → STYLE_SELECT → PLAYING (0/5)
                          │
                          ├─ 얼굴 고정 → PLAYING (1/5)
                          ├─ 눈 고정   → PLAYING (2/5)
                          ├─ 코 고정   → PLAYING (3/5)
                          ├─ 입 고정   → PLAYING (4/5)
                          └─ 머리 고정 → RESULT (5/5)
                                             │
                                             └─ RETRY → PLAYING (0/5)
```

- `INTRO`: `나의 초등학교 시절엔 어떤 친구가 있었을까?` 질문과 짧은 안내 표시
- `STYLE_SELECT`: 남학생 스타일, 여학생 스타일 또는 모든 파츠를 섞는 랜덤 선택
- `PLAYING`: 이전에 고정한 파츠 위에 현재 차례 파츠 하나만 변경되며 사용자의 입력을 기다리는 상태
- `LOCKING`: 한 번의 입력을 처리하며 현재 파츠에 짧은 고정 연출을 적용하는 상태
- `lockedCount`: 현재까지 고정된 파츠 수를 `0~5`로 관리
- `RESULT`: 닮은 친구 이름 입력, 이미지 다운로드, 다시 하기 제공

빠른 연속 터치로 두 파츠가 의도치 않게 고정되지 않도록 `LOCKING` 연출 중에는 버튼을 잠시 비활성화한다. 다시 하기를 누르면 `lockedCount`와 모든 고정 상태를 초기화한다.

## 4. 화면 및 UX 계획

### 모바일 우선 설계

모바일에서는 좌우 2단 화면을 억지로 축소하지 않고 단계별 전체 화면으로 전환한다.

#### 시작 화면

- 게임 로고보다 `나의 초등학교 시절엔 어떤 친구가 있었을까?`라는 질문을 가장 먼저 크게 표시
- 질문 아래에 `기억 속 얼굴 조각을 하나씩 멈춰 그 시절 친구를 만들어보세요.`라는 짧은 설명 표시
- `어떤 친구를 떠올려볼까?` 아래에 `남학생 모습`, `여학생 모습`, `랜덤으로 떠올리기 · 모든 파츠` 선택지를 표시
- 실제 개인정보를 입력하거나 전송하는 항목이 아니라는 점이 자연스럽게 드러나는 문구 사용
- 스타일 선택 버튼은 색상뿐 아니라 텍스트와 아이콘으로 구분
- 각 모드 카드 자체를 시작 버튼으로 사용하며, 별도의 시작 버튼 없이 한 번 누르면 바로 플레이 화면으로 이동
- 최근 저장 결과가 있으면 `내 앨범` 진입 버튼 표시

#### 플레이 화면

- 화면 중심에 최대한 크게 캐릭터 표시
- 파츠가 돌아가는 동안 가벼운 흔들림이나 CRT 느낌의 연출 적용
- 현재 고정할 파츠명과 `1/5~5/5` 진행도를 캐릭터 가까이에 표시
- 고정된 파츠에는 짧은 도장, 반짝임 또는 잠금 표시 연출 적용
- `얼굴만 → 얼굴+눈 → +코 → +입 → +머리` 순서로 파츠가 누적되어 완성 과정을 분명하게 보여준다.
- 하나의 `멈춤` 버튼을 엄지손가락으로 누르기 쉬운 화면 하단에 고정
- 버튼 라벨은 현재 차례에 따라 `얼굴 멈추기`, `눈 멈추기`처럼 변경
- 브라우저 하단 UI와 아이폰 홈 표시줄을 고려해 `env(safe-area-inset-bottom)` 적용
- 게임 중 불필요한 세로 스크롤 방지

#### 결과 화면

- 완성 캐릭터를 가장 먼저 노출
- `누구를 닮았나요?`라는 질문과 `친구 이름을 적어주세요` 입력란 표시
- 친구 이름은 앞뒤 공백을 제거한 뒤 최대 12자로 제한
- 이름을 입력하면 캐릭터 이미지 아래의 이름 영역을 실시간으로 갱신
- 다운로드 카드에는 `내 초등학교 친구`, 입력한 친구 이름, 얼굴 조합으로 정해진 결과명을 차례로 표시
- 친구 이름은 귀여운 손글씨 느낌의 한글 글꼴로 가장 크게 표시
- 이름이 비어 있으면 `이미지 다운로드` 버튼을 비활성화하고 짧은 안내 표시
- 버튼 우선순위: `이미지 다운로드` → `다시 만들기`
- 저장 결과는 앨범에 추가하거나 제거할 수 있게 구성

```text
┌────────────────────────┐
│                        │
│      완성 캐릭터        │
│                        │
├────────────────────────┤
│   내 초등학교 친구      │
│        민지            │  ← 귀여운 손글씨
│ 공책 속의 다정한 이야기꾼 │
└────────────────────────┘

누구를 닮았나요?
[ 친구 이름을 적어주세요       ]
[       이미지 다운로드       ]
[          다시 만들기         ]
```

#### 앨범 화면

- 저장된 조합을 썸네일 그리드로 표시
- 항목을 누르면 동일한 파츠 조합을 다시 렌더링
- 개별 삭제와 전체 삭제 제공
- 데이터가 없을 때는 게임으로 돌아가는 안내 표시

### 데스크톱 대응

- 왼쪽에 진행 중인 게임, 오른쪽에 최종 결과 영역을 배치한다.
- 결과가 확정되기 전 오른쪽에는 안내 문구 또는 실루엣을 표시한다.
- 좁은 화면에서는 자동으로 모바일형 단일 열 구조로 전환한다.

### 반응형 기준

- 최소 지원 화면 폭: `320px`
- 기본 설계 구간: `360px ~ 430px`
- 태블릿/데스크톱 전환 기준은 실제 디자인 시안에 맞춰 약 `768px` 전후로 결정
- 가로 모드에서도 버튼과 캐릭터가 잘리지 않도록 높이 기반 미디어 쿼리 추가
- 주요 터치 영역은 최소 `48 × 48px` 확보
- hover에 의존하는 기능은 만들지 않는다.

## 5. 비주얼 및 애니메이션 방향

### 테마 콘셉트

핵심 콘셉트는 **“공책 속 우리 반 친구 만들기”**다. 회상 시기는 **2000년부터 2010년까지의 한국 초등학교 시절**로 확정한다. 특정 브랜드를 그대로 복제하기보다, 공책에 볼펜으로 친구 얼굴을 조합해 그리던 놀이와 당시 교실·문방구·컴퓨터실에서 느낄 수 있었던 아날로그와 초기 디지털 감성을 함께 사용한다.

#### 감성 키워드

- 우리 반, 쉬는 시간, 짝꿍, 공책 낙서
- 필통, 연필, 지우개 자국, 자, 이름표
- 칠판, 분필, 알림장, 참 잘했어요 도장
- 문방구 스티커, 색종이, 캐릭터 편지지, 스티커 사진 꾸미기
- CRT에서 초기 LCD로 넘어가던 컴퓨터실, 볼 마우스, 픽셀 아이콘, 플래시 게임 화면
- 알록달록한 픽셀 글자, 반짝이는 웹 버튼, 로딩 게이지 같은 2000년대 초중반 디지털 장식
- 서툴지만 정감 있는 선, 조금 바랜 색, 손때 묻은 종이

#### 화면 모티프

- 배경은 줄공책이나 모눈종이처럼 구성한다.
- 주요 패널은 종이를 찢어 붙인 메모지, 이름표 또는 옛날 사진 카드 형태로 표현한다.
- 버튼과 진행 효과에는 2000년대 플래시 게임을 떠올리는 픽셀 아이콘과 단순한 입체 효과를 부분적으로 사용한다.
- 파츠가 고정될 때 연필 표시, 스티커 부착 또는 도장이 찍히는 듯한 피드백을 사용한다.
- 진행 표시는 `얼굴·눈·코·입·머리`가 적힌 알림장 체크 칸처럼 디자인한다.
- 결과 카드는 반·번호 같은 실제 개인정보 대신 가상의 캐릭터 번호와 긍정적인 별명을 사용한다.

#### 임시 색상 팔레트

| 용도 | 색상 | 값 |
| --- | --- | --- |
| 바랜 공책 종이 | 크림 | `#F3E7C5` |
| 볼펜 선과 본문 | 남색 | `#29486B` |
| 칠판 포인트 | 짙은 초록 | `#456557` |
| 선생님 도장과 강조 | 바랜 빨강 | `#C9544D` |
| 문방구 스티커 | 노랑 | `#F0C75E` |

색상은 레트로 분위기를 위한 초안이며, 실제 UI에서는 글자와 배경의 명도 대비를 우선한다.

#### 결과명 문체

결과명은 외모를 평가하거나 누군가를 놀리는 별명이 아니라, 학교생활의 즐거운 장면을 떠올리는 짧은 캐릭터 별명으로 만든다. 사용자가 적은 실제 친구 이름과 얼굴 조합으로 정해진 별명은 시각적으로 구분한다.

```text
내 초등학교 친구
민지
공책 속의 다정한 이야기꾼
```

결과명은 아래 세 묶음에서 한 항목씩 선택한 문장 구조를 사용한다. 모든 장면 문구를 `~의` 형태로 끝내 1,000개 조합이 자연스럽게 이어지게 한다.

| 학교 장면 10개 | 긍정적인 분위기 10개 | 친구 역할 10개 |
| --- | --- | --- |
| 우리 반의 | 신나는 | 웃음 요정 |
| 쉬는 시간의 | 다정한 | 놀이 대장 |
| 운동장의 | 유쾌한 | 응원 단장 |
| 급식 시간의 | 든든한 | 아이디어 박사 |
| 컴퓨터실의 | 야무진 | 척척 달인 |
| 문방구 앞의 | 씩씩한 | 추억 수집가 |
| 공책 속의 | 반짝이는 | 마음 지킴이 |
| 소풍날의 | 호기심 많은 | 이야기꾼 |
| 방과 후의 | 재치 있는 | 행복 메이커 |
| 모둠 활동의 | 명랑한 | 상상 탐험가 |

예: `쉬는 시간의 호기심 많은 아이디어 박사`, `운동장의 신나는 놀이 대장`, `공책 속의 다정한 이야기꾼`

실제 학교명, 실제 인물, 기존 캐릭터, 상표, 교표를 연상시키는 고유 요소는 사용하지 않는다.

### 아트 방향

- 첨부된 참고 이미지처럼 초등학생이 볼펜이나 사인펜으로 그린 듯 불완전하고 개성 있는 선
- 공책, 칠판, 문방구 스티커, 종이 질감, 2000~2010년 컴퓨터실과 플래시 게임 화면을 연상시키는 요소
- 과도하게 정교한 일러스트보다 파츠 조합의 엉뚱함이 잘 보이도록 제작
- 제한된 색상 팔레트로 전체 파츠의 통일감 유지
- 성별을 고정하는 색이나 장식을 피하고 서로 다른 파츠가 자유롭게 어울리게 제작
- 두 스타일을 단순히 파랑·분홍, 속눈썹 유무 같은 고정관념으로만 구분하지 않고 2000~2010년 학교 캐릭터의 다양한 머리와 표정을 참고
- 특정 얼굴 특징을 비하하거나 외모의 좋고 나쁨을 평가하는 표현은 사용하지 않음

### 파츠 제작 규격

- 모든 파츠는 같은 캔버스 크기와 기준점을 사용한다.
- 투명 배경 PNG 또는 SVG를 사용한다.
- 얼굴형을 기준으로 눈, 코, 입, 머리의 위치 좌표를 고정한다.
- 화면 표시용과 고해상도 결과 이미지 모두 같은 좌표 비율을 사용한다.
- 외부 출처 이미지가 아닌 직접 제작하거나 사용 권한이 확보된 에셋만 사용한다.

### 레이어 순서

기본 순서는 아래와 같이 정의하되, 머리카락이 얼굴 앞뒤를 모두 덮어야 한다면 머리 에셋을 뒤·앞 레이어로 분리한다.

```text
배경
→ 뒷머리
→ 얼굴형
→ 눈
→ 코
→ 입
→ 앞머리
→ 프레임/로고
```

### 애니메이션

- 파츠 변경 주기는 약 `80~180ms` 범위에서 조정한다.
- 모든 파츠를 동일한 프레임에 바꾸지 않고 서로 다른 간격을 사용한다.
- 멈춤을 누를 때마다 현재 차례의 파츠에 짧은 고정 연출을 적용하고, 다음 차례 파츠 하나만 나타나 움직인다.
- 다섯 번째 고정 연출이 끝난 뒤 전체 캐릭터 완성 효과와 함께 결과 화면을 보여준다.
- `prefers-reduced-motion` 사용자는 흔들림과 강한 전환 효과를 줄인다.

## 7. 데이터 설계

### 현재 결과 데이터

```js
{
  id: "result-1720915200000",
  createdAt: "2026-07-14T12:00:00.000Z",
  characterMode: "mixed",
  parts: {
    face: "boy-face-2",
    eyes: "girl-eyes-4",
    nose: "boy-nose-1",
    mouth: "girl-mouth-5",
    hair: "boy-hair-3"
  },
  title: "문방구 앞의 야무진 척척 달인",
  favorite: true
}
```

### localStorage 정책

- 키 이름: `schoolFriend:v1:results`
- 이미지 원본이나 Base64 문자열은 저장하지 않는다.
- 선택한 캐릭터 모드와 스타일 출처가 포함된 고유 파츠 ID만 저장하고 필요할 때 다시 렌더링한다.
- 결과명은 저장된 문자열을 신뢰하지 않고 파츠 ID 5개로 매번 다시 계산해 같은 얼굴과 이름의 대응을 유지한다.
- `characterMode`는 `boy`, `girl`, `mixed` 중 하나를 사용한다.
- 랜덤 결과를 복원할 때 스타일 출처를 잃지 않도록 단순 숫자 대신 `boy-face-2` 같은 전역 고유 ID를 사용한다.
- 실제 사용자의 성별이나 개인정보는 요청하거나 저장하지 않는다.
- 저장 개수는 1차 버전에서 최대 20개로 제한한다.
- 오래된 데이터부터 제거하거나 사용자에게 저장 한도를 안내한다.
- JSON 파싱 실패 또는 손상된 데이터가 있어도 앱이 실행되도록 예외 처리한다.
- 저장 데이터는 현재 브라우저와 접속 도메인에만 남으며 다른 기기와 동기화되지 않음을 안내한다.

### 닮은 친구 이름 입력 정책

- 이름 입력은 선택된 캐릭터 조합을 꾸며 다운로드하기 위한 용도로만 사용한다.
- 친구 이름은 현재 결과 화면의 메모리에만 두고 서버나 `localStorage`에는 저장하지 않는다.
- 앞뒤 공백과 줄바꿈을 제거하고 최대 12자로 제한한다.
- DOM 미리보기에는 `innerHTML`이 아닌 `textContent`를 사용하고 Canvas에는 `fillText()`로 그린다.
- 이모지나 지원되지 않는 문자가 들어오면 시스템 한글 글꼴로 대체해 빈 사각형이 나오지 않게 한다.
- 이름을 파일명에 포함하지 않아 개인정보 노출과 파일명 호환 문제를 줄인다.

## 8. 완성 이미지 다운로드

### 이미지 생성 방식

화면을 캡처하지 않고 별도의 고해상도 `<canvas>`에 선택된 파츠, 프레임, 결과명과 친구 이름을 순서대로 그려 PNG를 만든다. 휴대폰 화면 크기와 관계없이 결과물이 선명하게 유지되어야 한다.

친구 이름용 글꼴은 한글을 지원하고 웹 및 이미지 결과물 사용 권한이 확인된 파일을 프로젝트에 직접 포함한다. Canvas 렌더링 전에 `document.fonts.ready` 또는 `FontFace.load()`로 글꼴 로딩 완료를 확인한다. 이름이 길면 `measureText()` 결과에 따라 글자 크기를 줄여 정해진 영역 안에 맞춘다.

### 1차 버전 출력 규격

- 기본 크기: `1080 × 1080px`
- 형식: PNG
- 줄공책 또는 모눈종이 기본 배경 포함
- 이름표나 옛날 사진 카드 형태의 레트로 프레임 포함
- 상단 약 80%에는 완성 캐릭터, 하단 약 20%에는 이름 영역 배치
- 이름 영역에 `내 초등학교 친구`, 입력한 친구 이름, 긍정적인 결과명 포함
- 친구 이름은 귀여운 손글씨 느낌의 글꼴로 가장 크게 표시
- 파일명 예시: `school-friend-20260714-143052.png`

추후 인스타그램 피드용 `1080 × 1350px`, 스토리용 `1080 × 1920px` 선택 기능을 추가할 수 있다.

### 모바일 다운로드 순서

1. 이름 입력값을 검증하고 이미지와 글꼴 로딩 완료를 확인한다.
2. Canvas 결과를 `canvas.toBlob()`으로 PNG `Blob`으로 변환한다.
3. `URL.createObjectURL()`로 임시 URL을 만들고 `<a download>` 클릭 방식으로 저장을 요청한다.
4. 다운로드 시도 후 사용한 임시 URL은 `URL.revokeObjectURL()`로 해제한다.
5. 인스타그램 인앱 브라우저가 다운로드를 막으면 생성 이미지를 전체 화면으로 열어 길게 눌러 저장하도록 안내한다.
6. 길게 눌러 저장할 수 없는 경우에만 `Safari 또는 Chrome에서 열기` 도움말을 제공한다.

웹 공유 API와 앱별 전달 기능은 구현하지 않는다. 사용자는 다운로드된 PNG를 휴대폰의 사진 또는 파일 앱에서 원하는 방식으로 전달한다.

## 9. 인스타그램 인앱 브라우저 대응

- 첫 화면의 핵심 콘텐츠와 모드 선택 카드가 브라우저 UI에 가리지 않도록 한다.
- 사용자 에이전트만으로 기능을 단정하지 않고 실제 API 지원 여부를 검사한다.
- 다운로드 실패를 정상적인 상황으로 보고 이미지 길게 누르기와 외부 브라우저 안내를 제공한다.
- `localStorage` 기록은 인앱 브라우저 데이터 정리 시 사라질 수 있으므로 결과 이미지 저장을 적극 안내한다.
- 초기 로딩을 줄이기 위해 파츠 이미지를 압축하고 첫 게임에 필요한 에셋을 우선 로드한다.
- 게임 도중 네트워크가 끊겨도 이미 로드된 에셋으로 현재 플레이를 마칠 수 있게 한다.
- 외부 브라우저에서 열어야 할 때는 iOS와 Android에서 이해하기 쉬운 짧은 안내를 사용한다.

## 10. 접근성 및 사용성

- 모든 이미지와 버튼에 의미 있는 대체 텍스트 또는 접근 가능한 이름 제공
- 색상만으로 상태를 구분하지 않음
- 키보드의 Enter/Space로 시작, 5단계 멈춤, 다시 하기 가능
- 포커스 표시를 제거하지 않음
- 친구 이름 입력란에 항상 보이는 `<label>`과 현재 글자 수 안내 제공
- 이름이 너무 길거나 비어 있을 때 오류 원인과 해결 방법을 문자로 안내
- 작은 본문 글자와 낮은 대비를 피함
- 파츠 변경 중에도 `멈춤` 버튼의 위치가 움직이지 않게 유지
- 각 파츠가 고정될 때 현재 단계와 다음 단계를 `aria-live` 문구로 안내
- 빠른 깜빡임을 피하고 모션 감소 설정을 존중
- 결과 생성, 저장 성공, 실패 상태를 화면 문구로 명확하게 전달

## 11. 성능 계획

- 파츠 에셋은 표시 크기에 맞게 최적화하고 불필요하게 큰 파일을 사용하지 않는다.
- 친구 이름용 한글 글꼴은 필요한 굵기만 자체 호스팅하고 첫 결과 화면 전에 미리 로드한다.
- 초기 화면에 필요한 파일과 첫 플레이 파츠를 우선 로드한다.
- 게임 시작 전 이미지 디코딩을 완료해 파츠가 깜빡이지 않도록 한다.
- 애니메이션은 가능한 한 `transform`과 `opacity` 중심으로 구현한다.
- 반복 타이머는 결과 화면 전환 시 반드시 정리한다.
- Canvas 생성 후 사용한 Object URL은 `URL.revokeObjectURL()`로 해제한다.
- 느린 모바일 환경에서 초기 로딩, 게임 시작, 이미지 생성 시간을 실제 기기로 확인한다.

## 12. GitHub Pages 배포 계획

### 기본 배포 주소

```text
https://<github-username>.github.io/<repository-name>/
```

### Vite 경로 설정

프로젝트 페이지 주소를 사용할 경우 `vite.config.js`의 `base`를 저장소 이름에 맞춘다.

```js
import { defineConfig } from 'vite'

export default defineConfig({
  base: '/<repository-name>/',
})
```

사용자 루트 페이지 또는 개인 도메인을 연결하면 `base: '/'`를 사용한다.

### 자동 배포

- `main` 브랜치에 push할 때 GitHub Actions 실행
- Node.js 설치 및 `npm ci`
- `npm run build`로 `dist` 생성
- `dist`를 GitHub Pages artifact로 업로드
- GitHub Pages 환경에 배포
- 배포 실패 시 Actions 로그에서 빌드 및 경로 오류 확인

### 배포 전 확인

- 새로고침 후 모든 에셋 경로가 정상인지 확인
- GitHub Pages 하위 경로에서 이미지와 CSS가 404가 아닌지 확인
- 배포 주소에서 한글 글꼴 로딩과 PNG 다운로드가 정상인지 확인
- 배포 주소 변경 또는 개인 도메인 이전 시 기존 `localStorage`가 이어지지 않는다는 점 확인
- 저장 키는 같은 `github.io` origin의 다른 프로젝트와 충돌하지 않도록 고유한 접두사를 유지

- 이미 고정된 파츠는 이후 단계에서 변경되지 않는가

- 다섯 번째 고정 전에는 결과 화면이 열리지 않고, 다섯 번째 고정 후에는 정확히 한 번 열리는가
- 각 단계의 버튼 라벨과 `1/5~5/5` 진행 표시가 실제 상태와 일치하는가
- 다시 하기를 누르면 5개 파츠와 진행도가 모두 초기화되는가
- 각 캐릭터 스타일에서 부위별 파츠 5개가 모두 등장하는가
- 멈춘 조합과 결과 화면 및 다운로드 이미지가 일치하는가

- 입력한 친구 이름이 화면 미리보기와 PNG 하단에 동일하게 표시되는가
- 긴 이름의 글자 크기가 자동으로 줄어들어 프레임 밖으로 나가지 않는가
- 글꼴 로딩이 느린 환경에서도 PNG가 대체 글꼴로 잘못 생성되지 않는가
- 친구 이름이 `localStorage`나 파일명에 포함되지 않는가
- 결과를 저장한 후 새로고침해도 복원되는가
- 저장 항목 삭제와 전체 삭제가 정상 동작하는가
- 연속으로 빠르게 버튼을 눌러도 한 번에 두 파츠가 고정되거나 타이머가 중복 실행되지 않는가
- PNG 파일이 손상되지 않고 올바른 크기로 생성되는가

## 15. 1차 출시 완료 기준

- 남학생·여학생 캐릭터 스타일과 모든 파츠를 섞는 랜덤 모드를 선택할 수 있다.
- 각 스타일에 얼굴, 눈, 코, 입, 머리 파츠가 각각 정확히 5개씩 존재한다.
- 랜덤 모드에서는 각 부위의 파츠 10개가 성별 느낌과 관계없이 함께 순환한다.
- 모드 간 중복을 제외하고 총 100,000개의 고유한 시각 조합이 가능하다.
- 학교 장면·긍정적인 분위기·친구 역할을 조합한 결과명 1,000개가 제공된다.

- 남학생 모습과 여학생 모습에서도 각각 결과명 1,000개 전체에 도달하고, 결과명마다 2~5개의 시각 조합이 배정된다.

- 각 입력에서 `얼굴 → 눈 → 코 → 입 → 머리` 순서로 정확히 하나의 파츠가 고정된다.

- 모바일 320px 이상에서 핵심 UI가 잘리거나 겹치지 않는다.
- iOS Safari와 Android Chrome에서 게임을 끝까지 플레이할 수 있다.
- 친구 이름을 1~12자로 입력하고 결과 이미지 아래에서 확인할 수 있다.
- 입력한 이름이 허가된 한글 손글씨 글꼴로 다운로드 이미지에 합성된다.
- Android Chrome에서 PNG가 파일로 다운로드된다.
- 직접 다운로드가 제한되는 모바일 브라우저에서는 이미지 열기와 길게 눌러 저장 안내가 표시된다.
- Web Share API나 앱별 전송 기능 없이 다운로드 기능만 제공한다.
- 결과 PNG가 `1080 × 1080px`로 정상 생성된다.
- 저장한 파츠 조합이 새로고침 후에도 복원된다.
- GitHub Pages 배포 주소에서 이미지, CSS, JavaScript가 정상 로드된다.
- 로그인과 서버 없이 모든 1차 기능이 동작한다.

## 16. 위험 요소와 대응

| 위험 | 대응 |
| --- | --- |
| 인앱 브라우저가 파일 다운로드를 차단 | 생성 이미지를 열어 길게 눌러 저장하게 하고 필요 시 외부 브라우저 안내 제공 |
| 파츠 기준 좌표가 달라 얼굴이 어긋남 | 에셋 제작 전 공통 캔버스와 앵커 좌표를 먼저 확정 |
| PNG 합성 시 이미지가 Canvas를 오염시킴 | 동일 출처의 로컬 에셋만 사용하고 외부 이미지는 직접 그리지 않음 |
| localStorage 데이터 손상 또는 용량 초과 | 작은 조합 데이터만 저장하고 파싱·저장 예외 처리 |
| GitHub Pages에서 에셋이 404 | Vite `base` 설정과 상대 경로를 배포 전 검증 |
| 빠른 파츠 변경으로 저사양 기기에서 끊김 | 이미지 사전 디코딩, 적정 타이머 간격, 효과 단순화 |
| 레트로 장식 때문에 글자와 버튼이 읽기 어려움 | 장식 레이어와 정보 레이어를 분리하고 명도 대비 및 실제 기기 테스트 우선 |
| 캐릭터 별명이 외모 놀림이나 고정관념으로 느껴짐 | 외모 평가를 배제하고 행동·추억 중심의 긍정적인 결과명만 사용 |
| 남학생·여학생 스타일이 성별 고정관념을 강화함 | 실제 사용자 성별이 아닌 선택 가능한 캐릭터 스타일로 표현하고 색·표정·성격을 성별에 고정하지 않음 |
| 파츠 에셋이 25개에서 50개로 늘어 초기 로딩이 무거워짐 | 고정 스타일은 선택 세트만 로드하고, 랜덤 모드는 압축한 두 세트를 시작 전에 함께 프리로드 |
| 특정 기존 캐릭터나 문구류 디자인과 유사해짐 | 실제 상표, 교표, 캐릭터를 복제하지 않고 자체 제작한 일반적 학교 모티프만 사용 |
| 한글 글꼴이 로드되기 전에 PNG가 생성됨 | 다운로드 버튼 활성화 전에 글꼴 로딩 완료를 기다리고 실패 시 명시적 대체 글꼴 사용 |
| 긴 친구 이름이 이미지 영역을 벗어남 | 12자 제한과 `measureText()` 기반 글자 크기 자동 조정 적용 |
| 친구 이름이 개인정보로 불필요하게 남음 | 서버 전송, `localStorage` 저장, 파일명 삽입을 하지 않고 현재 결과 화면에서만 유지 |
